# 礼簿 Libu · 技术设计文档（v3）

> 配套文档：[产品方案](./产品方案.md)
> 关键决策：后端 **Python + FastAPI**；部署形态 **单家庭自用**；访问控制 **nginx Basic Auth**（无登录系统）。
> 数据模型：只记三类账 **`gift`(人情往来) / `child`(子女) / `parents`(孝敬父母)**，以 **事件(events)** 为容器批量记流水(records)，**往来对象(counterparties)** 支持个人/家庭与生命周期演变。
> 工程形态参考 `media_clawer`：**单 Docker 镜像**（前端 build 后由 nginx 静态托管，nginx + 后端同容器），数据库用 **SQLite(WAL)**。

---

## 1. 技术栈

| 层 | 选型 | 说明 |
|---|---|---|
| 反向代理 | **Nginx**（容器内） | 托管前端静态资源 + `/api` 反代到本机后端 |
| 前端 | **Vue 3 + Vite + TypeScript + Pinia** | 组件库 Naive UI；图表 ECharts；移动优先 + PWA |
| 后端 | **Python 3.12 + FastAPI** | Pydantic v2 校验；SQLAlchemy 2.0 ORM；Alembic 迁移 |
| WSGI/ASGI | **Gunicorn + Uvicorn worker** | 同 media_clawer 用 gunicorn 启动（`-k uvicorn.workers.UvicornWorker`）|
| 数据库 | **SQLite（WAL 模式）** | 单文件、零独立进程，契合单镜像；备份=拷贝一个 `.db` 文件 |
| 鉴权 | **nginx Basic Auth** | 站点级用户名密码（同 media_clawer，entrypoint 由 `APP_USERNAME`/`APP_PASSWORD` 生成 htpasswd）；**后端不做登录/JWT** |
| 部署 | **单 Docker 镜像** | 多阶段构建：node 构建前端 → python:slim 装 nginx，nginx + gunicorn 同容器（entrypoint 拉起）|

> v2 不再需要 APScheduler（周期账单已移除）。M2 的"还礼提醒"若要做，再按需引入进程内定时任务即可。

> **访问控制：无登录系统。** 内部家庭用，整站由 nginx Basic Auth 一个用户名密码保护；后端无 `users`/JWT/`/auth`。"记账人"改为前端**成员下拉**手选（见 `members` 表 §3.1），与鉴权解耦。

> 单家庭自用，刻意保持轻量：不引入 Redis/Celery/独立 DB 容器。整套系统 = **一个镜像 + 一个数据卷**（存放 `libu.db`）。

**为什么 SQLite 而非 MariaDB/PostgreSQL**：单家庭 2–4 人、写入低频（一天几笔），SQLite + WAL 完全够用；它无独立进程、无网络、无账号配置，与 media_clawer 的单镜像哲学完全一致，且备份/迁移只需拷贝一个文件，正好强化「数据可带走」的隐私卖点。只有要做多家庭公网 SaaS 或重并发写才需要 MariaDB。

## 2. 系统架构

参考 `media_clawer`：**所有东西打进一个镜像、跑在一个容器里**。前端在构建阶段产出静态文件，由容器内 nginx 直接托管；nginx 把 `/api/` 反代到同容器 `localhost:8000` 的 gunicorn(FastAPI)；数据落 SQLite 文件（挂数据卷持久化）。

```
┌─────────────────────────── 单容器（一个镜像）───────────────────────────┐
│                                                                        │
│   浏览器/PWA ──►  nginx :8080                                           │
│                   ├─ /        → /usr/share/nginx/html（前端 build 产物）│
│                   └─ /api/    → proxy_pass localhost:8000               │
│                                      │                                  │
│   （nginx 站点级 Basic Auth：先过用户名密码，再放行 / 与 /api/）       │
│                               gunicorn + UvicornWorker                  │
│                               FastAPI(:8000)（无登录，信任已过 nginx）  │
│                                      │                                  │
│                               SQLite  ./data/libu.db ──► 挂载数据卷      │
└────────────────────────────────────────────────────────────────────────┘
            entrypoint.sh：由 APP_USERNAME/APP_PASSWORD 生成 htpasswd → 起 nginx → exec gunicorn
```

请求流：浏览器 →（nginx Basic Auth 校验）→ 静态资源直出 / `/api` 反代 → FastAPI → SQLite 文件。
对外只暴露一个端口（参考项目用 `8080`）。**Basic Auth 是唯一访问控制**：前端 SPA 与 `/api` 都在 nginx `server` 作用域下被同一份用户名密码保护，后端不再校验身份。

## 3. 数据库设计（v3：事件 + 往来对象建模）

> v3 在「三类账」基础上引入两个一等公民：**`events`(事件/场合)** 与 **`counterparties`(往来对象)**。
> 核心工作流：**先建 event（收礼/送礼/宝宝/父母任一）→ 再在其下批量加 records（流水）**，呼应传统礼簿"办一场事、记一长串礼"的真实用法。
> 共 7 张表：`members`、`events`、`records`、`counterparties`、`counterparty_persons`、`tags`、`counterparty_tags`。
> 仍**不做**：账户/余额、转账、通用收入、预算、多账本、登录系统（访问控制交给 nginx Basic Auth）。

**通用约定**
- 主键 `INTEGER` 自增；`amount_cents` 整数存「分」；`records` 用 `deleted_at` 软删（单笔可撤销）。
- 三大主分类 `category ∈ {child, parents, gift}` 固定枚举、硬编码，不可增删。
- SQLite 开 **WAL 模式** + `foreign_keys=ON`（连接时 PRAGMA）。
- **统一删除策略（引用保护）**：`members`/`events`/`counterparties` 三个被引用的实体，**只要还有 record 引用就拒绝删除（返回 409）**，无引用才物理删除——等同 FK `ON DELETE RESTRICT`。"不想要了但要留历史"一律走隐藏/归档：成员/往来对象用 `is_active=false`，事件用 `is_closed=true`。只有底层流水 `records` 自身用软删（`deleted_at`）支持撤销。

### 3.1 members — 家庭成员（仅"记账人"下拉，**非登录账号**）
| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | **稳定不变**，record.member_id 永远引用它 |
| name | VARCHAR(50) | 显示名（爸爸/妈妈）|
| sort | INT | 下拉排序 |
| is_active | BOOL | 隐藏不删，保留历史引用 |
| created_at / updated_at | TIMESTAMP | |

**记账人生命周期：增 / 查 / 改 / 删**
- 新增 `POST /members`：加一个成员名（爸爸/妈妈/爷爷…）。
- 查询 `GET /members?include_inactive=`：默认仅 active，下拉用。
- 修改 `PATCH /members/{id}`：改名、调排序、`is_active` 显隐。改名后历史记录跟着显示新名（引用稳定 ID）。
- 删除 `DELETE /members/{id}`：**有 record 引用则拒绝删除**（返回 409 + 引用笔数），保护历史"谁记的"不断链；无引用才物理删除。**想从下拉去掉但保留历史 → 用 `is_active=false` 隐藏**，不要删。

### 3.2 events — 事件 / 场合（**新增，批量记账的容器**）
四种场景都先建一个 event，再往里批量加流水。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | |
| category | VARCHAR(12) | `child`(宝宝) / `parents`(父母) / `gift`(人情往来) |
| direction | VARCHAR(4) | 仅 `gift` 有意义：`in`(收礼) / `out`(送礼)；child/parents 固定 `out` |
| name | VARCHAR(80) | 事件名/事由：「李四婚礼」「我家宝宝满月酒」「春节给爸妈」「给娃报暑期班」 |
| occurred_at | DATE | 事件日期（其下 record 默认继承，可逐条覆盖）|
| note | VARCHAR(255) | 备注 |
| is_closed | BOOL | 结清/归档标记（归档后不可再加流水，可重开）|
| created_at / updated_at | TIMESTAMP | |

索引：`(category, direction, occurred_at)`。

### 3.2a 事件生命周期：增 / 查 / 改 / 删 / 归档

**新增** `POST /events`：选 category（+gift 选 direction）、填 name/日期/备注。建完即可在其下批量加流水。

**查询**
- 列表 `GET /events`：按 category/direction/日期范围/关键词筛选、分页；每条带**汇总**（笔数、合计金额；gift 事件可给收/送小计）。
- 详情 `GET /events/{id}`：事件信息 + 其下全部 records 明细 + 汇总。

**修改** `PATCH /events/{id}`
- `name`/`occurred_at`/`note`/`is_closed` 可自由改。
- 改 `occurred_at` 时，可选 `sync_record_dates=true`：把"仍沿用事件原日期"的 records 一并改到新日期（手动改过日期的不动）。
- **`category`/`direction` 受限**：一旦事件下已有 records，**不允许改**（records 已继承，改了会自相矛盾）；无 records 时才可改。需变更则删了重建。

**删除** `DELETE /events/{id}`（引用保护）
- **事件下有流水则拒绝删除**（返回 409 + 笔数/金额），防止误删一整场事的记录。
- 仅当事件**无流水**时物理删除。要删非空事件，须先逐条删/移走其下流水，再删空壳事件。
- **不想要了但保留历史 → 用归档 `is_closed=true`**，不要删。

**归档/重开** `is_closed`
- 归档（结清）：事件办完、礼记全了，置 `is_closed=true`，**禁止再往里加流水**（`POST /events/{id}/records` 拒绝）；台账/统计照常计入。
- 重开：置回 `false` 即可继续补记。

### 3.3 records — 流水（归属某 event，可批量创建）
| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | |
| event_id | INTEGER FK→events **NOT NULL** | 所属事件；**每笔流水必须挂事件**（强制）|
| category | VARCHAR(12) | 继承 `event.category`（冗余存便于直接筛选）|
| direction | VARCHAR(4) | 继承 `event.direction` |
| amount_cents | BIGINT | 金额（分）|
| subtype | VARCHAR(20) NULL | child→`supplies`/`education`/`medical`/`other`；parents→`cash`/`medical`/`health`/`other`；gift 一般不用（事由在 event 上）|
| counterparty_id | INTEGER FK→counterparties | 这笔记录对应的对象（**三类都必填**，应用层校验；类别须与事件一致）|
| member_id | INTEGER FK→members NULL | **记账人**（下拉手选）|
| occurred_at | DATE | 发生日期（默认 = event.occurred_at）|
| note | VARCHAR(255) | 备注 |
| deleted_at | TIMESTAMP NULL | 软删 |
| created_at | TIMESTAMP | |

索引：`(event_id)`、`(category, occurred_at)`、`(counterparty_id)`、`(member_id)`。

**校验规则**
- `event_id` **必填**：没有先建事件就不能记流水。`category`/`direction` 一律从所属 event 继承，禁止与 event 冲突。
- **三类统一**：所有 record 都**必填 `counterparty_id`**，且 `counterparty.category` 必须等于 `event.category`（父母事件只能选父母对象，子女事件只能选子女对象）。
- 子类 `subtype` 仅对「子女/父母」且 `direction=out`（送出/支出）有意义；人情往来与收入方向不分子类。
- **改事件归属**：`PATCH /records/{id}` 改 `event_id` 时，须重新继承新事件的 `category`/`direction` 并重校验对象类别。一般不鼓励跨事件移动。
- **快速记一笔**：UI 仍可"一步记账"，但底层会先建一个单条事件（事件名取自由输入或默认值），保证 `event_id` 始终存在 —— 模型上无"游离流水"。

### 3.4 counterparties — 往来对象（**新增，个人或家庭**，可演变）
支持"家庭维度"：一个往来对象可以是个人（张三）或家庭（张三&李四）。人情台账按它聚合。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | **稳定不变**，record 永远引用它 |
| category | VARCHAR(12) | **所属类别** `gift`/`child`/`parents`：人情往来对象、子女(老大/老二)、父母(男方/女方) |
| name | VARCHAR(80) | 显示名：个人「张三/老大」或家庭「张三&李四/男方父母」|
| kind | VARCHAR(10) | `person` / `household`（家庭）|
| relation | VARCHAR(30) | 关系（同事/亲戚/同学/朋友/邻居…）|
| note | VARCHAR(255) | 备注 |
| is_active | BOOL | 合并后被吞并的一方置 false（保留 ID 不真删，列表默认过滤）|
| created_at / updated_at | TIMESTAMP | |

**往来对象删除（引用保护）** `DELETE /counterparties/{id}`
- **有 record 引用则拒绝删除**（返回 409 + 引用笔数），保护人情往来历史不断链；无引用才物理删除（连带删其 persons/tags 关联）。
- **想从列表去掉但保留历史 → `is_active=false` 隐藏**；婚后并入某家庭 → 用 `merge`（§3.5a），不要删。

### 3.5 counterparty_persons — 家庭成员明细（household 用）
让「张三&李四」能拆成两个具体人名，便于按单人搜索、显示家庭构成、标注角色。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | |
| counterparty_id | INTEGER FK→counterparties | 所属往来对象 |
| name | VARCHAR(50) | 成员姓名（张三 / 李四）|
| role | VARCHAR(20) NULL | 可选：丈夫/妻子/本人… |

> 个人对象可不建明细；家庭对象登记 2+ 人，`name` 可由成员名自动拼成「张三&李四」或手填。

### 3.5a 往来对象生命周期：个人 → 家庭、合并（**关键能力**）

> **为什么零成本同步历史**：所有 record 引用的是稳定的 `counterparty_id`，不是姓名文本。改对象本身，历史 event/record 自动跟着走，无需任何数据迁移。

**场景一：张三结婚，个人升级为家庭，加入配偶李四（李四此前无往来记录）**
- 一次 `PATCH /counterparties/{张三id}`：`kind: person→household`、`name→"张三&李四"`、新增 `counterparty_persons`[张三, 李四(role=妻子)]。
- 张三名下**所有历史事件/流水原样保留**，台账即刻以「张三&李四」家庭维度展示。

**场景二：张三、李四婚前各自都有往来史，婚后合并为一个家庭**
- `POST /counterparties/{张三id}/merge` body `{from_id: 李四id, household_name:"张三&李四"}`：
  1. 把 `李四` 名下所有 `records.counterparty_id` 批量重指向 `张三id`（一条 UPDATE）；
  2. `李四` 的 person 明细并入张三的 `counterparty_persons`；标签合并去重；
  3. 目标对象 `kind→household`、`name→"张三&李四"`；
  4. `李四` 置 `is_active=false`（软隐藏，列表不再显示；其 records 已重指向张三）。
- 合并在单个事务内完成；合并后两人历史往来自动汇总到同一家庭。

> 反向操作（离婚拆分）非 MVP 需求，暂不做；如需，可后续支持把某些 person/record 拆回新对象。

### 3.6 tags / 3.7 counterparty_tags — 标签（多对多）
| tags | id PK · name VARCHAR(30) UNIQUE |
| counterparty_tags | counterparty_id FK · tag_id FK（联合主键）|

> 标签如「发小」「公司」「需重点维护」，用于筛选与分组人情往来。

### 预置子标签（seed，前端下拉，不入库为分类表）
- **子女 child**：用品(supplies)、教育(education)、医疗(medical)、其他(other)
- **孝敬父母 parents**：给钱(cash)、医疗(medical)、保健品(health)、其他(other)
- **人情往来 gift** 事由：放在 event.name 上（结婚/满月/乔迁/白事/过年…），record 不再单独存事由。

---

### 关系总览

```
events 1──* records *──1 counterparties 1──* counterparty_persons
   (一场事)   (一条礼/一笔开销)  (个人/家庭)        (张三、李四)
                  │                  │
                  *                  *──* tags（counterparty_tags）
                  │
                  1
               members（记账人）
```

**典型写入流程（收礼为例）**
1. 建 event：`{category:gift, direction:in, name:"我家宝宝满月酒", occurred_at:2026-06-20}`
2. 批量加 records：`[{counterparty_id:张三, amount:600}, {counterparty_id:"张三&李四", amount:800, member_id:妈妈}, ...]`
3. 人情台账按 `counterparty_id` 聚合：对「张三&李四」这一家，累计收到 800 / 送出 1200 / 净额 −400。

## 4. API 设计（REST / JSON，前缀 `/api/v1`）

统一返回 `{ "code": 0, "data": ..., "msg": "" }`。**无鉴权头**：请求到达后端时已过 nginx Basic Auth，后端不再校验身份（仅监听 `localhost:8000`，不直接对外）。

### 记账人 members（成员下拉，完整生命周期，详见 §3.1）
| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/members?include_inactive=` | 成员列表（默认仅 active，下拉用）|
| POST | `/members` | 新增成员名 |
| PATCH | `/members/{id}` | 改名 / 排序 / 显隐 |
| DELETE | `/members/{id}` | **有引用→拒绝(409)**；无引用→物理删除（去列表用 `is_active=false` 隐藏）|

### 事件 events（批量记账容器 + 完整生命周期，详见 §3.2a）
| GET | `/events?category=&direction=&from=&to=&q=` | 列表（每事件带汇总：笔数/合计/收送小计）|
| POST | `/events` | 新建事件（收礼/送礼/宝宝/父母）|
| GET | `/events/{id}` | 详情 + 其下 records 明细 + 汇总 |
| PATCH | `/events/{id}` | 改 name/日期/备注/归档(`is_closed`)；改日期可带 `sync_record_dates`；有流水时禁改 category/direction |
| DELETE | `/events/{id}` | **有流水→拒绝(409)**；空事件才物理删除（不想要保留历史用归档）|
| POST | `/events/{id}/records` | **批量加流水**：body 为 record 数组，统一继承事件 category/direction/默认日期（事件归档后拒绝）|

### 记账记录 records
| GET | `/records?event_id=&category=&direction=&counterparty_id=&from=&to=&member_id=&q=` | 分页+筛选 |
| POST | `/records` | 单条记账（**`event_id` 必填**；按 §3.3 校验）|
| GET/PATCH/DELETE | `/records/{id}` | 详情 / 编辑 / 软删 |

### 往来对象 counterparties（个人 / 家庭 + 关系/标签/备注 + 生命周期）
| GET | `/counterparties?kind=&relation=&tag=&q=` | 列表（按关系/标签/家庭维度筛选、搜人；默认仅 active）|
| POST | `/counterparties` | 新建（含 persons 家庭成员、tags）|
| GET/PATCH | `/counterparties/{id}` | 详情（成员明细/标签/往来汇总）/ 改 |
| DELETE | `/counterparties/{id}` | **有引用→拒绝(409)**；无引用→物理删除（去列表用 `is_active=false` 隐藏）|
| POST | `/counterparties/{id}/convert-to-household` | **个人→家庭**：改 kind、改名、加配偶 person（历史自动归入，§3.5a 场景一）|
| POST | `/counterparties/{id}/merge` | **合并**：body `{from_id, household_name}`，把 from 的 records 重指向本对象并升级为家庭（§3.5a 场景二）|
| GET/POST/DELETE | `/tags` · `/tags/{id}` | 标签管理 |

> `convert-to-household` 也可直接用 `PATCH /counterparties/{id}`（改 kind/name + persons）实现；单列一个语义化端点便于前端"TA结婚了"一键操作。`merge` 必须是独立端点（涉及跨对象 record 重指向 + 事务）。

### 人情往来台账（按往来对象聚合，含家庭维度）
| GET | `/gifts/ledger?relation=&tag=&q=` | 按 counterparty 聚合：每个对象（含「张三&李四」家庭）的 **送出 / 收到 / 净额 / 笔数 / 最近一次** |
| GET | `/gifts/ledger/{counterparty_id}` | 某对象全部往来明细（含所属事件，还礼时查证）|
| GET | `/gifts/by-event` | 按事件分布（哪场事收/送最多）|
| GET | `/gifts/by-relation` | 按关系分布（同事/亲戚…）|

### 统计概览
| GET | `/stats/overview?year=2026&month=` | 三类各自合计；人情往来 `in/out/净额`；可按年或按月 |
| GET | `/stats/trend?category=&months=12` | 某类的月度趋势（折线）|
| GET | `/stats/by-member?period=` | 成员记账占比 |

> 不再有 accounts / budgets / recurring-bills / import 接口 —— 已移除这些能力。

**后端关键实现要点**
- **金额**：全程整数分；Pydantic 出入参做 元↔分 转换或前端处理。
- **记账人**：写记录时带前端选中的 `member_id`（可空）；与鉴权无关，仅留痕。
- **三类校验**：按 §3.2 规则在 schema/service 层校验（非 gift 强制 out 且禁对方字段；gift 必填方向+对方）。

- **无身份校验**：后端只绑 `127.0.0.1:8000`，信任"已过 nginx Basic Auth"，不实现登录/JWT。

## 6. Nginx 配置要点（容器内，仿 media_clawer/nginx.conf）

```nginx
server {
    listen 8080;
    # 站点级 Basic Auth（唯一访问控制）：entrypoint 由 APP_USERNAME/APP_PASSWORD 生成
    # auth.conf；声明在 server 作用域，前端 SPA 与 /api 一并被保护。
    # 通配 include（auth.conf*）：文件缺失也不致 nginx 解析失败。
    include /etc/nginx/auth.conf*;

    location / {
        root /usr/share/nginx/html;          # 前端构建产物
        index index.html;
        try_files $uri $uri/ /index.html;    # SPA history 路由回退
    }

    location ^~ /api/ {
        proxy_pass http://localhost:8000/;   # 同容器内的 gunicorn(FastAPI)
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

> 安全提醒：Basic Auth 是明文凭据，公网访问务必叠加 HTTPS（前置 Caddy/Nginx 做 TLS，或仅在内网/VPN 使用）。后端 gunicorn 只绑 `127.0.0.1:8000`，不直接对外，确保流量必经 nginx 这道 Basic Auth。

## 7. Dockerfile（单镜像多阶段，仿 media_clawer）

```dockerfile
# 阶段1：构建前端
FROM node:18 AS frontend-build
WORKDIR /app
COPY frontend/ .
RUN npm install && npm run build

# 阶段2：后端 + nginx 同镜像
FROM python:3.12-slim
RUN apt-get update && apt-get install -y nginx && \
    apt-get clean && rm -rf /var/lib/apt/lists/*
# 前端产物交给 nginx
COPY --from=frontend-build /app/dist /usr/share/nginx/html
COPY nginx/nginx.conf /etc/nginx/conf.d/default.conf
RUN touch /etc/nginx/auth.conf          # 默认空=不启用 Basic Auth
# 后端
WORKDIR /app
COPY backend/ .
RUN pip install --no-cache-dir -r requirements.txt
COPY docker-entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh
RUN mkdir -p /app/data                  # SQLite 文件目录
EXPOSE 8080
CMD ["/entrypoint.sh"]
```

```sh
# docker-entrypoint.sh（要点，仿 media_clawer）
#!/bin/sh
set -e
# 从 APP_USERNAME/APP_PASSWORD 生成 Basic Auth（{SHA} 方案，无需 apache2-utils）
: > /etc/nginx/auth.conf
if [ -n "$APP_PASSWORD" ]; then
  python -c "import base64,hashlib,os;u=os.environ.get('APP_USERNAME') or 'admin';p=os.environ['APP_PASSWORD'];open('/etc/nginx/.htpasswd','w').write(u+':{SHA}'+base64.b64encode(hashlib.sha1(p.encode()).digest()).decode()+'\n')"
  printf 'auth_basic "Libu";\nauth_basic_user_file /etc/nginx/.htpasswd;\n' > /etc/nginx/auth.conf
fi
alembic upgrade head            # 或在 main.py 启动时 create_all + seed(默认成员)
nginx -g 'daemon off;' &
exec gunicorn app.main:app -k uvicorn.workers.UvicornWorker \
  --bind 127.0.0.1:8000 --workers "${GUNICORN_WORKERS:-2}" --timeout 120
```

> ⚠️ 若 `APP_PASSWORD` 为空则 `auth.conf` 为空文件 = **不启用 Basic Auth**（站点裸奔）。内部家庭部署务必设置 `APP_PASSWORD`。

**运行（单容器）**
```sh
docker build -t libu .
docker run -d -p 8080:8080 \
  -e APP_USERNAME=family -e APP_PASSWORD=设一个家庭密码 \
  -v libu-data:/app/data \           # 持久化 SQLite：/app/data/libu.db
  --name libu libu
```

> 与参考项目一致：**对外只一个端口 8080、只一个镜像、只一个数据卷**。访问站点时浏览器弹用户名密码框（Basic Auth）。SQLite 文件随 `libu-data` 卷持久化，备份=`docker cp` 或拷卷里的 `libu.db`。

## 8. 非功能性要求

- **安全**：站点级 nginx Basic Auth 作为唯一访问控制（务必设 `APP_PASSWORD`）；后端仅绑 `127.0.0.1` 不直接对外；公网访问叠加 HTTPS（前置 Caddy/在宿主做 TLS 终止）。Basic Auth 凭据明文，避免在不可信网络裸用。
- **备份**：SQLite 单文件，提供"导出 Excel/CSV"API + 直接拷贝 `libu.db` 两种方式，强化"数据可带走"卖点。
- **性能**：单家庭数据量小，SQLite(WAL) + 合理索引足够；gunicorn 2 worker 即可；统计接口可加内存缓存。
- **可维护**：Alembic 管理 schema 演进；后端类型注解 + 单测覆盖核心业务（三类校验、人情聚合、统计）。
