# 礼簿 Libu · 产品方案（PRD v3）

> 一本现代「礼簿」：中国家庭只记三笔最该被记住的账 —— **给孩子的、给爸妈的、和亲友来往的**。
> 自托管、注重隐私。技术形态：Nginx + 前端 + 后端打进**单个 Docker 镜像**，数据存 SQLite。

> 演进：v2 从"全功能家庭记账"收缩为**专项三类账**（砍掉日常流水、账户余额、收入、预算、多账本、账单导入）；v3 引入**事件(event)批量记账**与**往来对象(counterparty)**（个人/家庭、关系/标签/备注、生命周期演变），访问控制改为 **nginx Basic Auth**（无登录系统）。

---

## 1. 产品定位

**一句话定位**：不记柴米油盐，只记中国家庭围绕"人"的三笔账 —— 子女、父母、人情往来。

**名字由来**：`libu` = **礼簿**。传统礼簿是红白喜事上记份子钱的本子（谁随了多少、什么事、日后好还礼）。礼簿 Libu 把它数字化，并扩展到家庭里另外两笔同样需要长期回溯的"人情账"：给孩子的、给父母的。

**为什么这个形态成立**
- 日常小额开销（吃饭、购物、打车）没人爱记，记了也没用 —— 微信/支付宝账单本就能看。
- 但这三类开销是**大额、低频、需要日后回溯**的，恰恰没有顺手的工具：

| 这三笔账 | 痛点 | 别的工具为什么不行 |
|---|---|---|
| **人情往来** | 随了谁多少、收了谁多少，还礼要对得上 | 通用记账没有"对方/事由/收送方向"，更没有还礼视角 |
| **孝敬父母** | 给爸妈多少，兄弟姐妹分摊赡养要对账 | 淹没在流水里，捞不出来 |
| **子女** | 一年养娃到底花多少 | 散落在各处，没有汇总 |

**核心价值**：极简、聚焦、可回溯 + 隐私自托管 + 本土化。不追求"记全"，只记这三笔最该被记住的账。

---

## 2. 目标用户与场景

**主用户**：上有老下有小的家庭主力（28–45 岁），重视人情往来、需要对账/还礼/赡养分摊的人。

**典型场景**
1. 朋友结婚，随份子 800 → 记一笔【人情往来 · 送出 · 张三 · 结婚 · 800】。
2. 收到老家亲戚给孩子的满月红包 600 → 记一笔【人情往来 · 收到 · 二舅 · 满月 · 600】。
3. 给爸妈打生活费 2000 → 记一笔【孝敬父母 · 2000】，年底看"今年给了爸妈多少"。
4. 交孩子培训班 3000、买奶粉 400 → 记【子女 · 教育/用品】，看"今年养娃花了多少"。
5. 翻台账：当年随了李四 1000（他结婚），现在他孩子满月，该还礼 → 一查便知。

## 3. 核心建模：事件（Event）+ 往来对象（Counterparty）

礼簿的真实用法是"**办一场事、记一长串礼**"。所以记账以**事件**为单位组织：

> **工作流：先建事件 → 再批量加流水。** 四种场景都走这个流程。
> - 收礼事件：「我家宝宝满月酒」→ 批量记 张三 600、张三&李四 800、王五 500…
> - 送礼事件：「李四婚礼」→ 记随礼 800
> - 宝宝事件：「给娃报暑期班」→ 批量记 学费 3000、教材 200
> - 父母事件：「春节给爸妈」→ 批量记 给钱 2000、买保健品 500

### 3.1 子女（向下 · 纯支出）
**宝宝支出与孩子教育合并为一类**，覆盖从婴儿到学龄：
- 子标签（可选）：用品（奶粉/尿布/玩具/衣物）、教育（学费/培训班/早教）、医疗、其他。

### 3.2 孝敬父母（向上 · 纯支出）
给老人的钱与开销：
- 子标签（可选）：给钱/生活费、医疗、保健品、其他。

### 3.3 人情往来（向外 · 可收可送）—— 产品核心
数字化的礼簿。事件承载"事由/场合"，往来对象承载"和谁"：
- **方向**：送礼（随礼/给红包）/ 收礼（收礼/收红包）—— 由事件决定
- **往来对象（独立管理）**：
  - 可以是**个人**（张三）或**家庭**（张三&李四）—— 支持按家庭维度聚合展示
  - 带 **关系**（同事/亲戚/同学/朋友/邻居）、**标签**（发小/公司/重点维护）、**备注**
  - **会演变**：张三结婚 → 一键把"张三"升级为家庭"张三&李四"、加入配偶；**他名下所有历史往来自动归到这个家庭**，无需重记。若配偶此前也单独记过，可**合并**两人往来史。
- **核心能力——还礼视角**：按往来对象聚合，一眼看清跟某家/某人**送出多少、收到多少、往来净额**、最近一次往来，下次还礼不失礼、不吃亏。

> 严格只记这三类，**不设"其他"兜底**，不记日常流水（按你的要求）。

## 4. 功能范围与优先级

### MVP（M1，核心闭环）
- [ ] 访问控制：**nginx Basic Auth**（一个家庭用户名密码，内部用，无注册登录系统）
- [ ] 成员管理：设置里配几个成员名（爸爸/妈妈），记账时下拉手选"记账人"
- [ ] **往来对象管理**：个人/家庭（张三&李四）+ 关系/标签/备注；**个人→家庭升级、合并**（历史自动同步）
- [ ] **事件 + 批量记账**：建事件（收礼/送礼/宝宝/父母）→ 其下批量加流水（**流水必须挂事件**）
- [ ] 流水/事件列表：按三类、事件筛选，编辑、删除
- [ ] **人情往来台账**：按往来对象（含家庭）聚合送出/收到/净额，搜人、看全部往来
- [ ] 概览统计：本月/本年 三类各花多少、人情往来收支净额

### M2（增强）
- [ ] 报表：三类年度趋势、人情往来按事件/关系/标签分布、成员记账占比
- [ ] 还礼提醒：标记"待还礼"，或按历史金额给还礼参考
- [ ] 数据导出（Excel/CSV）、SQLite 文件备份
- [ ] 子标签/标签自定义、深色模式、PWA 离线记账

### 明确不做
- 日常流水（餐饮/购物/居家/交通…）、账户与余额、转账、通用收入
- **用户注册/登录、家庭成员邀请、账号权限**（改用 nginx Basic Auth）
- 预算、多账本、资产负债、账单导入、股票基金

## 5. 关键设计原则

1. **以事件批量记**：办一场事建一个事件，底下一长串礼/开销批量录入，符合礼簿真实用法；往来对象/事由可联想补全。
2. **可回溯优先**：统计围绕"按往来对象（含家庭）、按事件、按类、按年"回溯，服务于还礼与赡养对账。
3. **多人协作（轻量）**：一家人共享一本礼簿，每笔可标记账人（成员下拉）；不做账号体系，靠 nginx Basic Auth 一个密码进站；金额用整数（分）存储。
4. **隐私自托管**：单镜像 + 单 SQLite 文件，数据全在自己服务器，备份=拷一个文件。
5. **移动优先**：主要在手机上记，响应式 + PWA。
